The Core Difference

JS JavaScript - Errors at Runtime 
// This code runs... then crashes
function calculateTotal(price, quantity) {
  return price + quantity;
}

// Bug: passing string instead of number
const total = calculateTotal("100", 5);
// Returns "1005" not 105!
// No error shown until you check the result
TS TypeScript - Errors at Compile Time 
// TypeScript catches this BEFORE running
function calculateTotal(
  price: number,
  quantity: number
): number {
  return price + quantity;
}

// Error: Argument of type 'string'
// is not assignable to 'number'
const total = calculateTotal(100, 5);
// Returns 105 (correct!)

Common Bugs TypeScript Catches

Runtime Error 1. Type Coercion Bug

JavaScript

calculateTotal("100", 5)
Result: "1005" (string concatenation!)

TypeScript

calculateTotal(100, 5)
Result: 105 (correct math!)

Runtime Error 2. Null/Undefined Access

JavaScript

getUserEmail(null)
TypeError: Cannot read property 'email' of null

TypeScript

getUserEmail(user: User | null)
Compiler error: Object is possibly 'null'

Runtime Error 3. Misspelled Property

JavaScript

userProfile.emial
Result: undefined (silent failure)

TypeScript

userProfile.email // autocomplete!
Property 'emial' does not exist. Did you mean 'email'?

Runtime Error 4. Missing Function Arguments

JavaScript

greetUser("Alice")
Result: "Hello, Alice undefined!"

TypeScript

greetUser("Alice", "Smith")
Expected 2 arguments, but got 1

Runtime Error 5. Non-existent Method

JavaScript

numbers.flattern()
TypeError: numbers.flattern is not a function

TypeScript

numbers.flat() // autocomplete suggests!
Property 'flattern' does not exist. Did you mean 'flat'?

Side-by-Side Code Comparison

Function Definitions

JS JavaScript 
// No type information
function calculateTotal(price, quantity) {
  return price + quantity;
}

// No protection against null
function getUserEmail(user) {
  return user.email;
}

// Arguments can be anything
function greetUser(firstName, lastName) {
  return `Hello, ${firstName} ${lastName}!`;
}
TS TypeScript 
// Explicit types for parameters and return
function calculateTotal(
  price: number,
  quantity: number
): number {
  return price + quantity;
}

// Null safety with union types
function getUserEmail(user: User | null): string {
  if (user === null) return "No user";
  return user.email;
}

// Required arguments enforced
function greetUser(
  firstName: string,
  lastName: string
): string {
  return `Hello, ${firstName} ${lastName}!`;
}

Data Structures

JS JavaScript 
// Plain object - any shape allowed
const user = {
  name: "John",
  email: "john@example.com",
  age: 30
};

// Array can mix types
const data = [];
data.push({ name: "Alice", age: "25" }); // string age!
data.push({ name: "Bob" }); // missing age!
TS TypeScript 
// Interface defines exact shape
interface User {
  name: string;
  email: string;
  age: number;
}

const user: User = {
  name: "John",
  email: "john@example.com",
  age: 30
};

// Type-safe array
const data: User[] = [];
data.push({ name: "Alice", age: 25, email: "..." });
// Error if missing required fields!

Advanced TypeScript Features

TS TypeScript - Generics, Enums & More 
// Generics - reusable type-safe functions
function firstElement<T>(arr: T[]): T | undefined {
  return arr[0];
}

const num = firstElement([1, 2, 3]);    // type: number | undefined
const str = firstElement(["a", "b"]);  // type: string | undefined

// Enums - named constants
enum Status {
  Pending = "PENDING",
  Active = "ACTIVE",
  Completed = "COMPLETED"
}

const taskStatus: Status = Status.Active;

// Discriminated Unions - type-safe state handling
type Result<T> =
  | { success: true; data: T }
  | { success: false; error: string };

function handleResult(result: Result<User>) {
  if (result.success) {
    console.log(result.data.name); // TS knows data exists!
  } else {
    console.log(result.error); // TS knows error exists!
  }
}

When to Use Each

Use JavaScript When:

  • Small scripts or prototypes
  • Quick proof of concepts
  • Learning web development
  • Legacy codebases
  • Simple DOM manipulation

Use TypeScript When:

  • Large applications
  • Team collaboration
  • Long-term projects
  • Complex business logic
  • API integrations
  • Code that needs to be maintainable

TypeScript Mistakes & Best Practices

Learn from common mistakes and write better TypeScript code

Best Practice 1. Using any Excessively

Overusing any removes type safety, making your code prone to runtime errors.

❌ Bad

let data: any;
data = "Hello";
data = 42; // No type checking

✅ Good

let data: unknown;
data = "Hello";
if (typeof data === "string") {
  console.log(data.toUpperCase()); // Safe to use
}

Best Practice 2. Ignoring Strict Compiler Options

Not enabling strict mode weakens TypeScript's type checking.

❌ Bad

{
  "compilerOptions": {
    "strict": false
  }
}

✅ Good

{
  "compilerOptions": {
    "strict": true
  }
}

Best Practice 3. Not Using Type Inference

Explicitly typing everything, even when TypeScript can infer it.

❌ Bad

let count: number = 0;
let name: string = "John";

✅ Good

let count = 0; // TypeScript infers `number`
let name = "John"; // TypeScript infers `string`

Best Practice 4. Overusing Non-Null Assertions ( ! )

Using ! to assert non-null values without proper checks.

❌ Bad

let element = document.getElementById("myElement")!;
element.click(); // Risky

✅ Good

let element = document.getElementById("myElement");
if (element) {
  element.click(); // Safe
}

Best Practice 5. Not Handling undefined or null Properly

Ignoring potential undefined or null values.

❌ Bad

let name = user.profile.name; // Could throw an error

✅ Good

let name = user?.profile?.name ?? "Default Name"; // Safe

Best Practice 6. Misusing Enums

Using enums when a union type would suffice.

❌ Bad

enum Status {
  Active,
  Inactive,
}

✅ Good

type Status = "active" | "inactive";

Best Practice 7. Not Leveraging Utility Types

Manually creating types when utility types could simplify your code.

❌ Bad

interface User {
  id: number;
  name: string;
  email: string;
}

interface UserPreview {
  id: number;
  name: string;
}

✅ Good

type UserPreview = Pick<User, "id" | "name">;

Best Practice 8. Ignoring readonly for Immutability

Not marking properties as readonly when they shouldn't change.

❌ Bad

interface Config {
  apiUrl: string;
}
const config: Config = { apiUrl: "https://api.example.com" };
config.apiUrl = "https://malicious.com"; // Mutated

✅ Good

interface Config {
  readonly apiUrl: string;
}
const config: Config = { apiUrl: "https://api.example.com" };
// config.apiUrl = "..."; // Error

Best Practice 9. Not Using Generics Effectively

Writing repetitive code instead of using generics.

❌ Bad

function identityNumber(num: number): number {
  return num;
}
function identityString(str: string): string {
  return str;
}

✅ Good

function identity<T>(value: T): T {
  return value;
}

Best Practice 10. Ignoring interface vs type Differences

Using interface and type interchangeably without understanding their differences.

❌ Bad

type User = {
  name: string;
};
type Admin = User & { role: string };

✅ Good

interface User {
  name: string;
}
interface Admin extends User {
  role: string;
}

Best Practice 11. Not Using as const for Literal Types

Not preserving literal types when needed.

❌ Bad

const colors = ["red", "green", "blue"]; // Type: string[]

✅ Good

const colors = ["red", "green", "blue"] as const;
// Type: readonly ["red", "green", "blue"]

Best Practice 12. Not Handling Async Code Properly

Forgetting to handle promises or using any for async results.

❌ Bad

function fetchData(): Promise<any> {
  return fetch("/api/data");
}

✅ Good

async function fetchData(): Promise<MyDataType> {
  const response = await fetch("/api/data");
  return response.json();
}

Best Practice 13. Not Using Type Guards

Not narrowing types with type guards.

❌ Bad

function printValue(value: unknown) {
  console.log(value.toUpperCase()); // Error
}

✅ Good

function isString(value: unknown): value is string {
  return typeof value === "string";
}

function printValue(value: unknown) {
  if (isString(value)) {
    console.log(value.toUpperCase()); // Safe
  }
}

Best Practice 16. Not Using keyof for Type-Safe Object Keys

Accessing object keys without type safety can lead to runtime errors.

❌ Bad

function getValue(obj: any, key: string) {
  return obj[key]; // No type safety
}

✅ Good

function getValue<T, K extends keyof T>(obj: T, key: K) {
  return obj[key]; // Type-safe
}

Best Practice 17. Ignoring never for Exhaustiveness Checking

Not using never to ensure all cases are handled in a union type.

❌ Bad

type Shape = "circle" | "square";

function getArea(shape: Shape) {
  if (shape === "circle") {
    return Math.PI * 2 ** 2;
  }
  // Forgot to handle "square"
}

✅ Good

function getArea(shape: Shape) {
  if (shape === "circle") return Math.PI * 2 ** 2;
  if (shape === "square") return 4 ** 2;
  const _exhaustiveCheck: never = shape;
  throw new Error(`Unknown shape: ${shape}`);
}

Best Practice 18. Not Using Mapped Types

Manually creating similar types instead of using mapped types.

❌ Bad

interface User {
  id: number;
  name: string;
  email: string;
}

interface OptionalUser {
  id?: number;
  name?: string;
  email?: string;
}

✅ Good

type OptionalUser = Partial<User>;

Best Practice 19. Not Using satisfies for Type Validation

Not validating that an object satisfies a specific type.

❌ Bad

const user = {
  id: 1,
  name: "John",
  // Missing `email`
};

✅ Good

const user = {
  id: 1,
  name: "John",
  email: "john@example.com",
} satisfies User;

Best Practice 20. Not Using infer in Conditional Types

Not leveraging infer to extract types dynamically.

❌ Bad

type GetReturnType<T> = T extends (...args: any[]) => any ? any : never;

✅ Good

type GetReturnType<T> = T extends (...args: any[]) => infer R ? R : never;

Best Practice 24. Not Using Record for Dictionary-Like Objects

Using any or loose types for dictionary-like objects.

❌ Bad

const users: { [key: string]: any } = {
  "1": { name: "John" },
  "2": { name: "Jane" },
};

✅ Good

const users: Record<string, { name: string }> = {
  "1": { name: "John" },
  "2": { name: "Jane" },
};

Best Practice 25. Not Using Awaited for Unwrapping Promises

Not properly unwrapping nested promises.

❌ Bad

type Result = Promise<Promise<string>>; // Nested

✅ Good

type Result = Awaited<Promise<Promise<string>>>; // string

Best Practice 26. Not Using unknown for Catch Clauses

Using any for catch clauses, which can hide errors.

❌ Bad

try {
  // Some code
} catch (error: any) {
  console.log(error.message); // Unsafe
}

✅ Good

try {
  // Some code
} catch (error: unknown) {
  if (error instanceof Error) {
    console.log(error.message); // Safe
  }
}

Key Takeaways

  • Always prefer unknown over any for type safety
  • Enable strict mode in tsconfig.json
  • Use TypeScript's powerful utility types (Pick, Partial, Record, etc.)
  • Leverage type inference to reduce boilerplate
  • Use proper null/undefined handling with optional chaining (?.) and nullish coalescing (??)
  • Implement type guards for runtime type checking
  • Use readonly for immutability
  • Leverage generics for reusable, type-safe code
  • Use exhaustiveness checking with never type
  • Properly type async/await code

By avoiding these common mistakes and following best practices, you can write more robust, maintainable, and type-safe TypeScript code.
